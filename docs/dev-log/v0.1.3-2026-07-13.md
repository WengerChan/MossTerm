## 🏗️ 设计决策

### 1. 自己实现 vs 用 `x/crypto/ssh/knownhosts`

调研后发现 `x/crypto/ssh/knownhosts` 包的 API 限制：

```go
// 它的公开 API 只有：
func New(files ...string) (ssh.HostKeyCallback, error)
```

**不导出 DB 类型**，因此无法在 callback 命中"未找到"分支时手动 Add。我们需要的是"首次连接自动信任并持久化"，而它的 callback 在未知 host 时会直接返回 error。

**结论**：自己实现简化版，仅支持 MossTerm 实际需要的：
- host（精确匹配，无通配符 / 端口 / IP 范围）
- key type + base64
- 忽略 OpenSSH 的 marker（@revoked / @cert-authority）

未来要支持 OpenSSH 全兼容时，可以再换回标准库。

### 2. 三种 host key 策略的优先级

```go
switch {
case d.KnownHosts != nil:
    hostKeyCb = d.KnownHosts.HostKeyCallback()
case d.HostKeyCb != nil:
    hostKeyCb = d.HostKeyCb
default:
    hostKeyCb = ssh.InsecureIgnoreHostKey()  // ⚠️ MITM 风险（兜底）
}
```

v0.1.3 起 main.go 总会注入 KnownHosts，所以**实际不会走到兜底**。保留兜底是为了单元测试和"故意要走不安全模式"的场景。

### 3. 自动信任 vs 弹窗确认

v0.1.3 简化策略：
- **未知 host** → 自动 Add 写入文件 + 放行（与 OpenSSH `StrictHostKeyChecking=accept-new` 等价）
- **已知但 key 不匹配** → 拒绝（这是 MITM 信号，必须拒绝）
- **v0.2 计划** → 首次连接弹"是否信任"对话框，遵循 OpenSSH `ask` 模式

### 4. 路径策略

放在 config.toml 同目录下（`~/.config/mossterm/known_hosts`）：
- 与 OpenSSH 的 `~/.ssh/known_hosts` 隔离，便于排查
- 跟随 `XDG_CONFIG_HOME` / `$MOSSTERM_CONFIG` 环境变量
- 用户可以单独管理（`mossrm -clean-known-hosts` 等命令待 v0.2）

### 5. v0.1.2 修 bug 删错代码

`knownhosts.New` 在 v0.1.2 时删了一些行，结果把"文件不存在 → 创建空文件"的代码也删了。本次重写补回，并加测试覆盖。

## 🔄 数据流

```
ssh.Manager.Open(req)
  ↓
session.MemoryManager 注入 known_hosts (WithKnownHosts(kh))
  ↓
构造 connect.Deps{KnownHosts: kh}
  ↓
sshclient.New(deps) → Connector.knownHosts = kh
  ↓
Connector.Dial(params) 构造 ssh.ClientConfig
  ↓
cfg.HostKeyCallback = kh.HostKeyCallback()  // 智能 callback
  ↓
SSH 握手时：远端发 host key → callback 验证
  1. m.keys[host] 有且匹配 → return nil（放行）
  2. m.keys[host] 有但不匹配 → return ErrHostKeyMismatch（拒绝）
  3. m.keys[host] 没有 → m.Add(host, key, "mossterm-auto") + return nil
```

## 🛡️ 安全细节

### 已知风险（v0.1.3 接受）

1. **自动信任策略**：首次连接直接信任，**没有用户确认**。如果用户被 DNS 劫持连到攻击者机器，攻击者的 host key 会被写入 known_hosts。
   - **缓解**：v0.2 加 GUI 确认弹窗
   - **缓解**：v0.2 加"已信任 host 列表"展示

2. **不支持 OpenSSH 完整格式**：
   - 无通配符（`*.example.com`）
   - 无端口（`[host]:2222`）
   - 无 IP 范围（`192.168.1.*`）
   - 无 `|1|...` hash 编码
   - **缓解**：v0.2 评估要不要切到 `x/crypto/ssh/knownhosts`（届时通过 "trust on first use" + 持久化文件实现）

3. **多 key type 处理**：同一 host 可以有多个 key type（rsa + ed25519 + ecdsa）。当前实现一个 host 一个 key type。如果远端切换 key type（如 OpenSSH 升级），会被识别为"未找到"并自动添加新 key，**不会拒绝**。
   - **这个行为是合理的**：服务端升级到新 key 算法是正常情况，不应误判为 MITM

## 💡 关键经验

### 1. 标准库不一定够用
`x/crypto/ssh/knownhosts` 提供了基础校验但缺少"first-use trust + persist"能力。
遇到"标准库 + 我们需求不对齐"时，自己写简化版往往比强行扩展标准库更清晰。

### 2. v0.1.2 修 bug 时误删代码
`knownhosts.New` 在 v0.1.2 时手贱删了"创建空文件"的代码。本次 v0.1.3 补回 + 加测试覆盖。
**教训**：每次 commit 前 `go build ./...` + `go test ./...` 走一遍，CI 才能兜住。

### 3. ssh.HostKeyCallback 在 v0.22.0 改签名
- 旧：`func(...) bool`（true = 放行）
- 新：`func(...) error`（nil = 放行）
- v0.1.1 写的 `func(...) bool` 现在是错的，但 Go 编译器通过（type alias）—— 实际行为：return false = 拒绝，return true = 放行，但 ssh 库对 bool 类型不会调……
- 实际：v0.1.1 ~ v0.1.2 用的是 `connect.HostKeyCallback = ssh.HostKeyCallback` alias，签名跟着 ssh 走，是返回 error 的。我们的 callback 必须返回 error。
- **v0.1.3 第一次写错了（返回 bool），第二次改对**

### 4. 测试数据：不要硬编码私钥
最初我在测试里硬编码 `sampleED25519Key`，结果 `ssh: short read` 失败（base64 padding / 换行不严格）。
改用 `crypto/ed25519.GenerateKey + ssh.NewSignerFromKey` 动态生成，干净可靠。
