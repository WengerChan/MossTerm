# v0.6.1 · 端口转发补丁 + SFTP 下载双入口

> 收 v0.6.0 三个已知遗留：
> 1. **ctx watcher goroutine leak**（3 处）—— per-conn 取消传播 watcher 在 `context.Background()` 下永久泄漏
> 2. **Remote forward 集成测试误判 skip** —— 实际代码早就能跑，验明 + 保留 PASS 状态
> 3. **SFTP 下载双入口** —— 双击文件 + 顶部 Download 按钮（FileList 替换原 toast "v0.2 实现"）

## 关键设计决策

### 1. ctx watcher leak 修复

- **根因**：`local.go:132` / `remote.go:118` / `socks5.go:135` 三处 per-conn watcher 写的是 `go func() { <-ctx.Done(); close(conn) }()`。当调用方传 `context.Background()` 时 `ctx.Done()` 返回 nil chan，`<-nil` 永远阻塞 → **每条 conn 泄漏一个 goroutine**
- **修法**：`select { case <-ctx.Done(): ... case <-t.stopCh: ... }`，任一信号触发回收
- 兼容旧 API：`t.stopCh` 是 `baseTunnel` 已有的 close 信号，Stop 触发时也会关 watcher

### 2. Remote forward 集成测试

- **B 子任务报告有误**："Remote skip" 是子任务 B 自己加的安全判断，实际 `TestTunnel_Remote` 已写好且 in-process server 早就有 `tcpip-forward` + `forwarded-tcpip` handler
- 验证：`go test -race -run TestTunnel_Remote -count=1 ./internal/tunnel/` → PASS（0.05s）
- 这次没改 Remote 逻辑，只在 dev-log 里去掉"skip"标记

### 3. SFTP 下载双入口

- **FileList**：原"双击文件"是 toast "v0.2 实现"占位，改成调 `onDownload(sessionId, ~/Downloads/<name>, remotePath)` 真正下载
- **SftpPanel 顶部**：原本已 import `<Download>` icon 但没渲染，加 `<Button icon={<Download />} onClick={downloadAll}>` —— 批量下载当前目录所有文件
- **`startDownload` 闭包**：复用 SftpBrowserContent 上传那套形态 —— 调 `App.StartDownload` → `useTransferStore.getState().upsertJob` 占位 → 后续 `transfer:progress` 事件由 useTransferEvents 接管
- **顺手修 v0.6.0 遗留**：`SftpPanel` 底部传输任务列表用旧字段（`j.size` / `j.transferred` / `j.speed`）—— v0.6 UploadJobInfo 改名 `totalBytes` / `bytesSent` / `speedBps`，没跟上；改字段 + `jobs` 来源从 useSftpStore stub 切到 useTransferStore 真数据
