# v0.5.0 · SFTP + 多 tab + 首次信任 GUI（2026-07-14）

> **三个并行子任务** + **集成修复** —— SFTP 客户端、多 tab 拆分 UI、首次连接信任弹窗，
> 三件事一起做一起合。中间修了几个测试 bug + 一个"历史性 build 失败"。

## 三个并行子任务交付

### A · SFTP 客户端（`internal/sftpclient`）

**目标**：把 v0.0 留的 8 个 stub 方法替换成真实 `github.com/pkg/sftp` 实现。

**实现要点**：
- `Open(*ssh.Client, ...Option)` 包装 `sftp.NewClient`
- `List` / `Stat` / `ReadDir` / `Open` / `Read` / `Write` / `Close` / `Remove` / `Rename` / `Mkdir` / `Rmdir` 全部接 pkg/sftp
- `Close()` 后所有方法返回 `errors.New("client closed")`，**不** panic（关键）
- 11 个单元测试（`entryFromFileInfo` 5 个 + `Close`/nil-safe 3 个 + Option 2 个 + 签名 1 个）
- 真实分页 / 断点续传 / 并发安全**不**在 v0.5.0 范围（接口里 `pageSize` / `NextToken` 保留占位）

**设计决策**：
- `pageSize/NextToken` 接口占位：pkg/sftp `ReadDir` 不支持分页，v0.5.0 一次性返回全量；
  后续接真实分页协议时**方法签名保持不变**，只换实现
- `*Client` 接收器不防御 nil（Go 惯例，让 nil deref 暴露 bug 而非 silently swallow）
- `ctx` 参数保留但不强制（SFTP 协议无 cancel 机制）—— 跟上层 ctx-aware 调用约定对齐

**取舍**：
- 不加互斥锁：pkg/sftp 自己的 client 是 goroutine unsafe，**加锁责任在调用方**（wailsbindings 层）
  —— 保持包职责单一，不假装是并发安全的

### B · 多 tab + 拆分 UI（`frontend/src/components/tabs`）

**目标**：替代单 session 标签页，支持多个 tab 横向排列 + 同一个 tab 内左右/上下分屏。

**实现要点**：
- Pane 树 = **二叉树**：节点要么是 Split（带 direction + 两棵子树）要么是 Leaf（带 sessionId）
- `splitPane(tabId, paneId, direction, newSessionId)`：在指定 pane 位置插入 Split 节点，
  原 sessionId 保留到第一个子节点，新 session 放第二个
- `closePane(tabId, paneId)`：删除 pane，若父 Split 只有一个孩子则"上提"（collapse），
  若 Split 变空则连父 Split 一起删
- Terminal 组件 mount 时 guard `sessionId !== null` —— 避免空 pane 渲染崩溃
- Zustand store 集中管理 tabs/panes 树 + active 状态

**代码 review only**（无 npm build 环境）—— 逻辑与 v0.0 既有 SessionTree / useTerminal 集成 OK。

**取舍**：
- 不做 N-ary 拆分（>2 子树）：二叉树够用，且 UI 操作（方向键 / drag）天然对应二叉
- 不做 pane 拖拽换位：v0.5.0 范围之外

### C · 首次信任 GUI 弹窗（`internal/knownhosts` + `internal/ui/wailsbindings` + `frontend/.../TrustRequestModal`）

**目标**：连接陌生 host 时不再"自动信任 + 写 known_hosts"，而是弹 Wails 模态框让用户决策。

**架构关键**：避免循环依赖

```
known_hosts.HostKeyCallback  →  EventEmitter.EmitTrustRequest(ctx, TrustRequest)
                                ↓
                          Wails 事件总线  "knownhosts:trust-request"
                                ↓
                          TrustRequestModal 监听 → 渲染 → 用户点 trust/reject
                                ↓
                          App.TrustHost(requestID, action)  // wails binding
                                ↓
                          Manager.ReplyTrust(requestID, action)  // 写 trustReplyCh
                                ↓
                          HostKeyCallback select 收到 reply → trust → Add + 放行 / reject → 拒绝
```

**设计要点**：
- **`EventEmitter` 接口在 known_hosts 包内定义**（`EmitTrustRequest(ctx, TrustRequest)`），
  不 import wails runtime —— 核心层可独立单测
- **`wailsEmitter{}` 在 main.go 零字段 struct**，同时实现 `app.EventEmitter` 和
  `knownhosts.EventEmitter` 两个接口 —— 避免在 main.go 维护两份 emitter 适配器
- `EmitTrustRequest` **非阻塞**（emit 后立即返回，**不**等 reply）—— 等待 reply 走
  `trustReplyCh` 另一条路径（select + 60s 超时）
- `trustReplyCh` 容量 1 + 5s `ReplyTrust` 超时：保证无挂起请求时立即返回 + 异常不阻塞
- ID 严格匹配：拿到"别人的 reply"或"过期 reply"时返回 ID mismatch error，让 SSH 握手失败
  （不静默放行）
- `len(ev.Data) > 0` 门控 sub:overflow 累加（与 v0.2.3 同样的"防递归 drop 通知"设计）

**已知限制**（v0.5.0 接受，v0.6+ 优化）：
- `trustReplyCh` 容量 1：并发 SSH 连接到不同未知 host 时只有第一个能拿到正确 reply，
  其余会 timeout —— 实际场景（用户在 modal 里点之前不会同时点第二次连接）下不影响
- 真正的并发 trust 场景需要 `map[requestID]chan TrustReply`，留给 v0.6+

**TrustRequestModal 行为**：
- 监听 `knownhosts:trust-request` 事件 → 渲染 host / keyType / fingerprint
- 三个按钮：Trust（信任 + 持久化） / Reject（拒绝） / 拒绝 1 次（临时）
- 即使 `App.TrustHost` 返回 error 也强制关 modal（让 SSH 连接走 timeout 失败路径）
- 失败弹 toast 提示，不阻塞 UI

## 集成修复（主理自己干）

### 1. known_hosts 测试逻辑 bug × 2

**(1) `TestHostKeyCallback_TrustRequest_Timeout`**：原测试同步调 `waitForCall` 在 `cb()`
**之前**，但 emit 是 `cb()` 内部发出的，2s 内根本没信号 → "no Emit within 2s"。

修法：把 `waitForCall` 放到 goroutine 里监听 emit，主 goroutine 才调 `cb()`。

**(2) `TestReplyTrust_AfterRejection_StateRecovers`**：`mockEmitter.waitForCall` 固定返回
`e.calls[0]`，多次 emit 后第二次拿到的还是第一次的（过期）ID，导致 HostKeyCallback
收到错 ID 的 reply → ID mismatch error。

修法：把 `mockEmitter` 改成 **FIFO 队列**（每次 `waitForCall` pop 最早一条），跟
"第 N 次 emit 触发第 N 次 waitForCall"的自然测试节奏对齐。

### 2. session 测试稳定性：`TestOpen_AsyncReturnsBeforeDial`

**根因**：v0.2.0a 起 `readLoop` 在 reader 退出（dataCh 关闭）时会自动调
`sessionImpl.Close`，把 state 推到 Closed。原 `stubSession.Read` 立即返回 `io.EOF`
→ reader 立即退出 → readLoop 立即 Close → 100ms 观察窗口里 `Info().State` 已经是
Closed 而不是 Established（之前是 flake 的，取决于机器快慢）。

修法：`stubSession.Read` 改成**阻塞**（`<-s.readDone`），`Close` 时 close `readDone`。
这才符合真实 SSH session 的行为（一直阻塞等远端数据）。

10 次 + race detector 跑都稳。`stubSession` 注释也补充了"v0.2.0a 后 readLoop 自动 close"
的设计动机，避免后人改回 `io.EOF` 又踩坑。

### 3. `go build` 历史性失败（v0.0 就有，但没人触发）

**根因**：`cmd/mossterm/main.go` 里 `//go:embed frontend/dist` 是相对 main.go 的，
要找 `cmd/mossterm/frontend/dist/`，但这个目录**从来不在 git 里**。v0.0 时代的
`/tmp/mossterm-bin` 7.5MB 是 `wails build` 出的，**裸 `go build` 从 v0.0 起就跑不通**。
v0.1.1 修的 "embed 路径相对 main.go" 是把"绝对路径"改成"相对 main.go"，但没创建目标
目录。

修法：在 `cmd/mossterm/frontend/dist/index.html` 放占位文件（说明在 HTML 注释里），
让 `go build` 能过。`wails build` 会覆盖这个占位。

`.gitignore` 也加规则保留这个占位（防止被 `dist/` 通用规则误 ignore）：

```
!/cmd/mossterm/frontend/dist/
!/cmd/mossterm/frontend/dist/index.html
```

**未来避免再踩**：
- AGENTS.md / Makefile / CI 应该明确：`go build` ≠ `wails build`；前者只验 Go 编译，
  后者才是完整 webview 打包
- 想用 `go build` 验"代码至少能编译"，必须有占位 `cmd/mossterm/frontend/dist/`

## 反思

- 三个 sub-task 都能并行做，但**集成阶段必须 main agent 自己干**：测试 bug 修复、
  embed 路径修复这些是"上下文密集 + 范围小"的事，分给 sub-agent 反而成本更高
- v0.0 时代的 `go build` 失败是**历史包袱**，AGENTS.md 之前没文档化"go build
  ≠ wails build"的区别，导致每次都踩坑。这次顺手把 `cmd/mossterm/frontend/dist/`
  占位 + .gitignore 规则做了，下个 session 不会被坑
- A 的 SFTP 分页简化（一次性返回）是合理取舍 —— pkg/sftp 不支持分页，强行
  自己实现分页（用 Open+Readdir 模拟）成本不值，等 v0.6+ 看实际大目录场景再说
