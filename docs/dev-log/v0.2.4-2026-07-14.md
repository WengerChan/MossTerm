## 🔑 关键设计决策

### 1. `publishMu` 锁的范围（最小化）

- **只保护**"state 转换 + 状态事件发布"的原子性
- **不保护** `tryPublish` 自身（lock-free，hot path 性能不变）
- **不与** `subMu` / `connMu` 嵌套

### 2. 三方法分层

```
setStateAndPublishIf(expect, to)  // CAS 路径：dialInBackground 用
  → publishMu.Lock → CAS + tryPublish → Unlock

setStateAndPublish(st)            // 无条件路径：Close 用
  → publishMu.Lock → store + tryPublish → Unlock

setStateAndPublishLocked(st)      // 内部 helper：调用方已持 publishMu
```

### 3. Close 路径 IO 步骤移出 publishMu

顺序：
```
publishMu(Closing) → release
connMu + IO + signalDone
publishMu(Closed) → release
subMu 关闭 sub channels
```

**保证** Closing 事件在 s.done 关闭**前**已进 events 通道（fanoutLoop drain 能拿到）。

### 4. 锁顺序文档化（防死锁）

- `publishMu` ⊥ `subMu`（Close 路径上严格不嵌套）
- `publishMu` ⊥ `connMu`（同上）
- `tryPublish` 不持任何锁

## 🐛 v0.2.1 标记但未修的 race

v0.2.1 doc comment 写过：
> "CAS 成功之后 Close 抢先会让 stale event 通过。Subscriber 需丢弃'event.state 早于 Info().State'的事件。完全消除需加 publishMu 锁，Close 也得拿。"

**v0.2.4 完成**：`publishMu` 锁 + Close 持锁 + 序列化 setState + tryPublish。

stale event race 已消除。
