# v0.5.10 · SFTP streaming upload

> **GB 级文件支持**：v0.5.3 的"readAsArrayBuffer → SftpUploadFile"一次性上传
> 改成"分片 + 流式 + 进度 + 断点续传"四件套。默认 4 MiB chunk / 2 路并发。
> 10 GiB 硬上限保护，UI 实时进度可取消。

## 关键设计决策

1. **SFTP WriteAt 是并发正确路径**：pkg/sftp `*Client` 并发安全，`*File` **不是**
   → worker 用 `WriteAt(buf, offset)` 各写自己区间；`sftpclient.ReadWriteCloser` 加 `io.WriterAt`（pkg/sftp `*File` 已实现）
2. **Truncate 预分配**：upload 前先 SETSTAT 让 server 端分配磁盘空间，让"磁盘满"在 WriteAt 之前暴露
3. **Manifest 持久化**：`<configDir>/transfers/<id>.json`，atomic-rename 写盘，每片完成 flush 一次
4. **Resume 校验**：local 文件 mtime/size 不一致 → `ErrLocalChanged` 拒绝续传（避免数据错位损坏）
5. **进度节流 200ms**：避免 1 GiB 文件 256 次回调刷屏
6. **race-free 设计**：每 worker 独立 sha256 hasher + per-chunk 槽位 + 主 goroutine 单写者累加 manifest
7. **ctx 取消**保留 manifest：已传 chunks 留作 Resume 凭证
8. **10 GiB 大文件保护**：双保护 OOM + 远端磁盘
9. **drag handler 改走 StartUpload**：v0.5.3 的 `SftpUploadFile`（100 MiB cap）binding 保留但前端不用

## 踩坑（race detector + 集成测试 抓出）

| 问题 | 修复 |
|---|---|
| sha256 hasher 共享 → race | 每 worker 独立 `sha256.New()` |
| worker 直改 manifest → race | worker 只发 `chunkDone{index}` 到 ch；主 goroutine 单写者累加 |
| emitLoop `for range manifestCh` 死等 close | 改 non-blocking drain |
| `InMemHandler` 写后立即读死锁 | 写完关 client → 重 dial 新 SFTP client → 再读 |
| `InMemHandler` 不自动建父目录 | `MkdirAll(parent)` + best-effort 忽略 `os.ErrExist` |
| Wails ctx 不能外部注入 value → sessionID 走 req 字段 | `UploadRequest.SessionID` + wailsbinding 注入 ctx |
| `transfer.Progress` 命名冲突 v0.1 stub | rename `Direction` / `EngineProgress` |
| `t.Context()` Go 1.24+ 新 API | 改 `context.Background()` + `t.Cleanup(cancel)` 保兼容 |

## 关键 API

```
// 后端
Upload(ctx, uploader, req, manifestDir, progress) error

// Wails bindings
StartUpload(ctx, req) (string, error)        // 返回 transferID
CancelUpload(ctx, transferID) error
ListTransfers(ctx) []JobInfo
GetTransfer(ctx, transferID) (JobInfo, bool)

// Wails 事件
transfer:progress  → 200ms 节流
transfer:done      → 全部 chunk 完成
transfer:error     → 失败 / 取消
```
