# v0.6.3 · 真实 sftp-server 集成测试

> 把 v0.5.10 streaming 集成测试的 "pkg/sftp in-process SFTP server" 路径扩到
> "**真实 OpenSSH sftp-server binary**" 路径：`internal/testutil/sftpd` 包 +
> 一条 1MB smoke 集成测试。InMemHandler 测试保留（100MB 性能测试不重跑）。

## 关键设计决策

### 1. testutil/sftpd 设计

- **`Start(t, opts) -> *SFTPD`**：起 in-process `ssh.Server` + 真实 sftp-server 进程；sftp-server 走 stdin/stdout 通过 SSH channel 桥接（与 v0.6 tunnel.local.go 双向 io.Copy 同款）
- **跨平台 binary 路径**：darwin `/usr/libexec/sftp-server` / linux `/usr/lib/openssh/sftp-server` 兜底 `/usr/libexec/sftp-server` / windows `t.Skip`
- **sftp-server -d 行为**：v0.6.3 实测发现 `-d` **只 chdir 不 chroot**（"foo.txt" 落 WorkDir；"/foo.txt" 走真实文件系统根）—— 测试必须用**相对路径**，绝对路径会越界。已在 sftpd 包 doc + 新测试注释里写明
- **退出死锁规避**：t.Cleanup LIFO 顺序——先注册 ssh 后注册 sftp，结束 sftp-server 收 stdin EOF 先退，ssh 后关

### 2. 跟 v0.5.10 集成测试并存

- **保留** InMemHandler 5 个测试（1MB / 50MB / 100MB / 并发 / 断点续传）—— 快 + 协议层覆盖
- **新增** `TestUpload_Integration_RealSFTPServer`：1MB smoke，只验"真实 binary 跑通"，不复跑 100MB（sftp-server 子进程 IO 比 InMemHandler 慢 ~10x）
- windows runner 没 sftp-server → sftpd.Start 内部 `t.Skipf`（不 Fatal），CI 那条线 skip 不 fail

### 3. 跟 sftpd 的握手死锁陷阱

- 写完 → 显式关原 sftp client → 新建独立 ssh conn → 新建 sftp client → 读
- sftpd 报告里说的 "Cleanup LIFO 顺序" 印证了这套（先 ssh 后 sftp，结束 sftp 先退）

## 关键 API

```go
package sftpd

type Options struct {
    WorkDir      string  // 空 → t.TempDir()
    User         string  // 默认 "test"
    Password     string  // 默认 "x"
    ServerBinary string  // 空 → 按 GOOS 选
    LogStderr    *bool   // nil → 默认 true
}

type SFTPD struct { /* unexported */ }
func Start(t *testing.T, opts Options) *SFTPD
func (s *SFTPD) HostPort() (host string, port int)
func (s *SFTPD) User() string
func (s *SFTPD) Password() string
func (s *SFTPD) Close()
```
