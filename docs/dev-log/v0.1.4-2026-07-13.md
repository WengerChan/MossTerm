## 🔧 关键设计决策

### 1. keepalive 失败处理：不主动关 client

只 log + 退出协程。理由：
- 关闭 `*ssh.Client` 会同时挂掉所有走该 client 的 session / channel
- session 关闭走 `sess.Close() → conn.Close()` 自然链路
- 下次 `SendRequest` 因 conn 关闭返回 err → 协程自动退出

### 2. SendRequest 超时：goroutine + select + Timer

x/crypto v0.22.0 的 `ssh.Client` 没有自己的 `SendRequest`，走嵌入的 `Conn` interface：
```go
type Conn interface { SendRequest(name string, wantReply bool, payload []byte) (bool, []byte, error) }
```
**3 个返回值，无 context**。原 spec 假设的 `SendRequest(ctx, ...)` 是 v0.33+ 签名。

用 goroutine + select + `time.NewTimer` 实现超时控制。代价：timeout 触发时启动 SendRequest 的 goroutine 短暂泄漏（最坏 TCP 重传超时，分钟级）。v0.33+ 升级到 ctx 版可彻底解决。

### 3. `done` per-Connector（不是 per-Dial）

单次 `Connector.Close()` 关掉该实例下所有 keepalive。per-session 关闭走"下次 SendRequest 失败"路径（最多多跑一个 interval）。

### 4. `Close()` 不在 `connect.Connector` 接口里

有意的：`session.Manager` 持的是 `connect.Connector` interface，看不到 `Close`；keepalive 资源由 sshclient 自治，零侵入接口契约。

## 📈 安全 / 健壮性提升

| 场景 | v0.1.3 | v0.1.4 |
|---|---|---|
| 长 idle 30 分钟 | TCP 被中间设备单方面断开，下次操作才感知 | 每 30s 发送 keepalive，连接不断 |
| 网络断连 | 永远不感知，靠 SSH 协议层下次 I/O 失败 | 3s 内通过 keepalive 失败感知 |
| 资源泄漏 | 无 keepalive 协程 | closeOnce 保证 Close 幂等，done 通道阻止泄漏 |
