## 🔑 关键设计决策

### 1. **自实现 vs 标准库**

调研 x/crypto v0.22.0 `ssh/knownhosts`：
- `NewDB` **不存在**（仅有 `New(files...)` 返回 callback）
- DB 类型未导出

**结论**：必须自实现匹配算法。`wildcardMatch` 直接 port 自 x/crypto `wildcardMatch`，行为与 OpenSSH `addrmatch.c` 一致。

### 2. matchHost 算法（OpenSSH 兼容）

- **端口必须完全相等**（`example.com:22` 不匹配 `example.com:2222`）
- **`*.example.com` 不匹配 `example.com`**（基域 ≠ 子域）
- **`*` 匹配任意字符序列（含空）**
- **`?` 匹配任意单字符**
- **`192.168.1.*` 匹配整个 IP 段**
- **`[host]:port` 端口格式**

### 3. 数据结构改造

```go
// v0.1.3: map[host]map[type]b64
// v0.2.2: []entry{patterns []pattern, key}
type entry struct {
    patterns []pattern  // 多 pattern 行共享一个 key
    keyType  string
    keyBase64 string
}
type pattern struct {
    host string
    port string
}
```

通配符匹配不能 hash key，必须线性扫 entries。

### 4. 向后兼容

- 旧 `~/.config/mossterm/known_hosts` 文件**不用迁移**
- 公共 API 4 个外部调用点签名 + 语义不变
- 3 个行为变化（更严格 / 更安全 / 增强）

## 🐛 主理总集时修的 bug

`wildcardMatch("*", "")` 返回 false，测试期望 true。

**根因**：循环开头先判 `len(strBytes) == 0 → return false`，没机会走到 `*` 分支。

**修复**：把 `*` 分支移到循环开头最先处理，并合并连续 `*`：

```go
for {
    // 先处理 '*' —— 它能匹配空串
    if len(patBytes) > 0 && patBytes[0] == '*' {
        for len(patBytes) > 0 && patBytes[0] == '*' {
            patBytes = patBytes[1:]
        }
        if len(patBytes) == 0 {
            return true  // 末尾全是 '*'，匹配任何剩余
        }
        // 尝试 patBytes 切 strBytes 的某个前缀
        for j := 0; j <= len(strBytes); j++ {
            if wildcardMatch(string(patBytes), string(strBytes[j:])) {
                return true
            }
        }
        return false
    }
    // ... 其他分支
}
```

## ⚠️ 已知边界（v0.5+ 计划）

- **大文件性能**：O(n) 扫 entries；1000+ 条目开始变慢。v0.5+ 改 trie
- **HASH 编码（`|1|...`）永远不匹配**，首次连会被 auto-trust 写明文。后续从 x/crypto port `newHashedHost`
- **Negation pattern `!host`** 不识别（当字面处理）
- **Callback 入参 host 解析失败 → return error**（v0.1.3 是 return nil，**更安全**）
