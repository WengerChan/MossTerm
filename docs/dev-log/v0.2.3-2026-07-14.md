## 🔑 关键设计决策

### 1. 防 sub:overflow 递归的核心：`len(ev.Data) > 0` gate

**问题**：sub:overflow 事件本身被 broadcast 给 sub——如果某 sub 队列还是满的，sub:overflow 事件本身被 drop，要不要递归 emit？

**解法**：`newSubOverflowEvent` 构造的 Event `Data` 字段为空。broadcast default 分支加 `if n := len(ev.Data); n > 0` 守卫——sub:overflow 事件被 drop 时**不**累加、不置位 pending。**递归在物理上不可能发生**。

这比"加状态机分支"简洁一个数量级——把不变量编码进数据本身。

### 2. maybeEmitSubOverflow 与 maybeEmitOverflow 完全镜像

两者结构、并发安全论证、`bytes > 0` 才 emit 三个细节全部对齐。区别仅常量名 + 访问的 atomic 字段。

### 3. fanoutLoop emit 顺序：先 data overflow 后 sub overflow

```go
s.broadcast(ev)
s.maybeEmitOverflow()      // data overflow 先
s.maybeEmitSubOverflow()   // sub overflow 后
```

**理由**：data overflow 内部 broadcast 一个 overflow 事件，这个事件也可能被 sub drop——但因为 Data 空，**不会**触发 sub drop 累加。顺序无关，但要保持"两个计数器在同一轮 fanoutLoop 迭代里都被处理"。

### 4. 累加条件 `len(ev.Data) > 0` 双重意义

1. 正面：data 事件包含用户可见字节流
2. 反面：state/exit/error/overflow/sub:overflow 的 Data 都为空——元事件丢失不累加

## 📊 v0.2.0 vs v0.2.3 对比

| 维度 | v0.2.0 (data overflow) | v0.2.3 (sub drop) |
|---|---|---|
| 触发点 | `tryPublish` events 通道满 | `broadcast` 单个 sub channel 满 |
| 瓶颈含义 | 后端 readLoop > 后端 fanout | frontend consumer < 后端 fanout |
| 事件类型 | `overflow` | `sub:overflow` |
| 前端文案 | "server too fast" | "client too slow" |
| 累加字段 | `overflowBytes` | `subDropBytes` |

---

## ⚠️ 已知边界

- **sub drop 是"观测"，不"限速"**（v0.5+ 加 backpressure）
- **多 sub drop 累加不区分 sub**（当前用例单 sub 够用，v0.5+ 按 sub 分）
- **sub:overflow 自身被 drop 时信息丢失**（可接受，元数据不影响 UI）
