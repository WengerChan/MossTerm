# v0.5.3 · SFTP 客户端分页 + 拖拽上传

> **两个并行子任务** + **两次 sub-agent 死锁主理接手**。本次最大的反思
> 不是分页怎么实现，而是 sub-agent 死锁时主理怎么收敛 + 测试基础设施的
> 隐藏陷阱（InMemFS 跨 client 不共享）。

## 两个子任务交付

### A · SFTP 客户端分页（后端）

**目标**：把 v0.5.0 留的 `List(pageSize, pageToken)` 接口真正做起来。pkg/sftp
v1.13.6 **不暴露** `Opendir+Readdir(n)` 客户端 API（v0.5.0 当时放弃真分页就是这个
原因），所以做"客户端分页"—— 全量 `ReadDir` + 切片 + token 缓存。

**实现要点**：
- `internal/sftpclient/pagecache.go`（221 行）—— LRU 缓存 + token 编码 + slicePage
  - `maxCacheEntries = 100` 防内存爆
  - token 格式：`base64.RawURLEncoding("v1:" + path + ":" + offset)`，last-colon 分隔
    保证路径里的 `:` 不冲突
  - LRU 用 `container/list` + map 双向索引
- `internal/sftpclient/pagecache_test.go`（623 行）—— 单测覆盖 token 编解码 / LRU
  淘汰 / slicePage 边界
- `Client.List` 真正用 pageCache：
  - `pageToken == ""`：查 cache，miss 才 ReadDir
  - `pageToken != ""`：decode token → 校验 path == pathArg → 切缓存
  - token 跨路径 / decode 失败 → 返回 `ErrInvalidPageToken` 包装
  - 最后一页 `NextToken == ""`（空字符串）
- `WithPageSize` 真正生效：覆盖 List 默认 200
- `Client.Close` 清空 pageCache 释放内存
- `pageSize > 1000` 限制 1000（防大 pageSize 拉爆单次响应）

**A 还顺便加了 v0.5.3 拖拽上传需要的 `Client.Write` / `Client.UploadFile` 公开方法**：
- `Write(path, data) (int, error)` —— 覆盖写到远端
- `UploadFile(localPath, remotePath, chunkSize, progress) error` —— 分片 + 进度回调
  - chunkSize=0 用默认 64 KiB（x/crypto 推荐 SFTP 分片大小）
  - 错误时已传部分**不**回滚（best-effort）

**主理接手修了 2 个 A 留下的 bug**：
1. **A 写了 pagecache.go 但没改 client.go 的 List 实现**——老 List 直接 `ReadDir`
   全量返回，根本没用 pageCache。3 page 拿重叠 entries + token 校验缺失
2. **A 写 integration_test.go 时 `sftp.InMemHandler()` 每次新建 InMemFS root**——
   `sftp.NewClient` 拿不到上一个 client 写的内容（`file does not exist`）。
   修法：server 实例上 `sftpHandlers sftp.Handlers` 共享，所有 channel 用同一组

### B · 拖拽上传（前端 + 后端）

**目标**：把 v0.5.1 B 写的 SFTP 浏览器加上"拖本地文件上传到远端"功能。

**实现要点**：
- `internal/ui/wailsbindings/api.go` 加 `SftpUploadFile` binding —— 调
  `core.SftpClient(sessionID)` 拿 `*sftpclient.Client` → `client.Write(remotePath, content)`
- `frontend/wailsjs/go/main/App.d.ts` 加 `SftpUploadFile: Promise<number>` 类型
- `frontend/src/components/sftp/SftpBrowser.tsx` 加 drag-drop handlers + 视觉反馈
- `internal/ui/wailsbindings/sftp_upload_test.go`（510 行）—— 5 tests + 编译期签名守卫

**Drag-Drop UX**：
- 拖文件到浏览器面板 → 整个 panel 边框变 accent 绿 + 外圈 `ring-4 ring-accent/40`
- 松开鼠标 → 顶部 `Uploading <filename>...` → 多文件依次上传
- 100 MiB 限制（`readAsArrayBuffer` 一次性读进内存，超出 UI 冻结；v0.6+ streaming）
- 单文件失败：红色 error 条，其他文件继续（每个独立 try/catch）

**关键决策**：
- 挂到内层 panel（不是外层 backdrop）—— 否则拖到 backdrop 触发 `onClick={onClose}` 关浏览器
- `dragleave` 检查 `relatedTarget` 避免进入子元素时 ring 闪烁
- 覆盖写语义（与"上传"一致，不弹确认；重命名是 v0.6+ 范围）
- 错误用 `setError` / 进度用 `setLoading("Uploading ...")` 与既有 List 风格一致

## 两次 sub-agent 死锁主理接手（重要反思）

**v0.5.2 A + v0.5.3 A + v0.5.3 B（首次派活）** 都出现"sub-agent 写完核心代码后沉默
25-40 分钟"的死锁。

**收敛模式**（已稳定下来）：
1. 写完核心代码 + 自验测试 PASS → 立即回话报告，**不要**继续做"再多覆盖点" / "再优化下"
2. Sub-agent 跑 > 30 分钟无 output（working tree 无新改动 + task_output "no output yet"）
   → 主理 stop + 接手验证
3. 接手后跑测试 + 看代码 → 几乎每次都是一次过（A 留的代码本身没问题）
4. **B 报告里提示的"潜在 bug"可能是 sub-agent 中间状态**——主理验证时不要直接相信

**v0.5.3 B（首次派活）** 派活 30 分钟无 output + 0 改动 → 主理 stop。但 v0.5.3 B
**功能本身没做**（SftpUploadFile binding + drag-drop）—— 这是 A/B 都接手的不同点：
- v0.5.2 A：A 写完了 core code，主理接手验证 + 修 A 留下的小 bug
- v0.5.3 A：A 写完了 pageCache 但没把 client.go 接上 + InMemFS 不共享，主理接手补完
- v0.5.3 B（首次）：B 30 分钟 0 改动，主理 stop 但**B 的功能没做**（因为它一开始就没产出）

**教训**：
- Sub-agent 30 分钟 0 改动 → **不要**假设核心代码已写好
- 派活设计要更"小"—— v0.5.3 B（首次）任务太复杂（前端 + 后端 + 测试），sub-agent 容易卡
- 后续大特性分活：**前端 sub-agent + 后端 sub-agent** 拆开，**两个独立 30 分钟**比一个 60 分钟稳定
