### B · known_hosts 并发 trust 完整实现

**目标**：兑现 v0.5.0 写的"留给 v0.6+ 优化"承诺 —— 用 `map[requestID]chan TrustReply`
替换 v0.5.0 的 `trustReplyCh chan TrustReply`（容量 1）。

**实现要点**：
- `Manager.trustReplyCh` → `Manager.trustMu + Manager.trustWaiters map[string]chan TrustReply`
- `HostKeyCallback`：per-request `replyCh := make(chan TrustReply, 1)`，注册到 map，defer 清理
- `ReplyTrust`：用 requestID 查 map 找对应 ch，5s 兜底超时
- **不需要 ID mismatch 防御**（replyCh 一一对应，ID 必匹配）

**关键设计**：
1. **per-request channel > 共享 channel**：replyCh 一一对应，每个 callback 自己的 channel
   收自己的 reply
2. **独立 `trustMu`（不复用 `m.mu`）**：`m.Add` 持 `mu` 期间做文件 IO，不应被 trust map
   阻塞；两条路径完全解耦
3. **defer 删 waiter**：trust / reject / timeout 任何路径都通过 defer 清理 map，无泄露
4. **不开 goroutine**：SSH handshake 本身就是同步的，开 goroutine 反而增加复杂度

**行为差异**：

| 场景 | v0.5.0 | v0.5.2 |
|---|---|---|
| 5 并发 SSH trust 5 不同 host | 第一个成功，**其余 timeout** | **全部成功**，互不干扰 |
| ReplyTrust 无 waiter | buffered 默默吸收（不报错） | 立即 error |
| ReplyTrust 错 ID | 默默成功 + 错 ID mismatch | 立即 error |
| ID mismatch 防御 | 必要（replyCh 共享） | **不需要**（replyCh 一一对应） |

**测试覆盖**：
- `TestHostKeyCallback_ConcurrentTrustRequests` —— 5 goroutines × 5 hosts all trust
- `TestHostKeyCallback_ConcurrentTrustRequests_MixedActions` —— trust / reject 混合
- `TestReplyTrust_UnknownID_ReturnsError` —— 错 ID 立即 error
- `TestReplyTrust_NoPendingRequest_ReturnsError` —— 名字终于和语义对齐（新行为真返回 error）
- ❌ 删除 `TestHostKeyCallback_TrustRequest_IDMismatch`（v0.5.0 设计缺陷）→ 改成 `t.Skip()` 桩
