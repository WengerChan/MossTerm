## 🔑 关键设计决策

### 1. `Start` 调用时机——选"Open 同步阶段调"（自创 Option C）

**用户给的选项**：
- A：拆 `Start`（检查） + `startIOLoops`（启动 IO）
- B：保持 `Start` 启动 IO，但 manager 在 `SetDialedSess` 完成后调

**小弟选的 C**：`Start` 在 `Open` 同步阶段就调，3 个 loop 在 Open 返回时都跑起来。`readLoop` / `writeLoop` 在 `waitForSess` 阻塞等 `s.sess`；`fanoutLoop` 立即可用，state 事件能立即 broadcast。**所有 Connecting → Authenticating → Established 状态转换都能被 subscriber 看到**。

### 2. 状态机 race 防御——用 `setStateIf`（基于 `atomic.CompareAndSwap`）

异步化引入新 race：
```
goroutine A (dialInBackground): state=Connecting
goroutine B (Close):           setState(Closing) → setState(Closed)
goroutine A:                   setState(Authenticating) ❌ 覆盖 Closed
```

`setStateIf(expect, to)` 把"检查+设置"合并成原子 CAS。

### 3. 失败时不自动 Close

`dialInBackground` 失败路径只 `setStateIf + tryPublish + signalDone`，**不调 Close**。caller 决定何时清理。

### 4. 顺手修的 v0.1.x bug

- **`Info().State` 状态缓存**：v0.1.x 把 State 缓存在 Info 里，状态转换后 `Info().State` 仍是初始 `Connecting`——前端永远看不到 `Authenticating` / `Established`。v0.2.1 改为每次读从 `s.state.Load()` 拉。
- **`waitForSess` StateFailed 死循环**：原来只在 `StateClosed` 退出，dial 失败会让 readLoop / writeLoop 永远轮询。

## ⚠️ 已知边界（v0.2.x 计划）

- **stale event**：CAS 成功之后 Close 抢先会让 out-of-order 事件通过。Subscriber 需丢弃"event.state 早于 Info().State"的事件。完全消除需加 `publishMu` 锁。
- **dial ctx 取消**：Close 不会取消 Open 传入的 ctx。caller 应传带超时的 ctx。
