## 🏗️ 设计决策

### 1. 注入路径：依赖反向注入

```
wailsbindings ─→ app ─→ session.MemoryManager.WithSecrets(sec)
                                  ↓
                       m.Open 构造 connect.Deps{Secrets: sec}
                                  ↓
                          sshclient.Connector.secrets
                                  ↓
                          c.authMethods(PublicKeyAuth{KeyID})
                                  ↓
                          c.loadSigner(keyID) → secrets.Get → bytes
                                  ↓
                          loadSignerFromBytes → ssh.Signer
```

**为什么用反向注入而不是直接 import**：
- `connect` 包保持协议无关
- `sshclient` 不污染业务层（业务层用 `session.AuthSpec`）
- `secret.Store` 是接口，测试可注入 mock

### 2. 公开 publickey 路径：Signer 与 KeyID 二选一

```go
type PublicKeyAuth struct {
    Signer     ssh.Signer  // 优先：调用方已解析好
    KeyID      string      // 备选：从 secret.Store 拉
    Passphrase string
}
```

两种使用方式：
- **直接用 Signer**：测试 / 高级用户 / agent 转发的场景
- **走 KeyID**：业务层 Profile 配置的常见路径，sshclient 自动从 secret.Store 拉

### 3. publickey 解析放在 sshclient 层

不让 `connect` 包 import `sshclient`（避免循环）。改：
- `connect.AuthMethodFromSpec` 只做"形态转换"（kind 字符串 → AuthMethod 类型）
- `sshclient.authMethods` 在拿到 `PublicKeyAuth{KeyID, Signer==nil}` 时，**自己做 signer 解析**
- `loadSigner` 用 `c.secrets.Get` 拉私钥，调 `loadSignerFromBytes` 解析

### 4. signerCache 保留

`Connector.signerCache` 是个 LRU（cap 64），按 keyID 索引 Signer：
- 第一次 Open 解析后写入缓存
- 后续 Open 同一 keyID 命中缓存，省一次 secret.Store.Get
- 安全性：signer 本身不存明文私钥，只存已解析的 ssh.Signer（内存中）

## 💡 关键经验

### 1. 注入 vs 直接 import
SSH auth 这种"协议实现 + 业务凭据"组合，**用依赖注入**比直接 import secret 包干净：
- 保留 connect 包的协议无关性
- 方便单测（mock secret.Store）
- 未来要换密钥来源（HashiCorp Vault / 1Password CLI）只换注入的接口实现

### 2. 公开输入 / 内部处理分工
- `connect.AuthMethodFromSpec`：做"形态转换"，不碰 secret
- `sshclient.authMethods`：做"运行时解析"，能拿到 connector 注入的 secrets
- 边界清晰，各司其职

### 3. sum-type + 默认实现的灵活性
`PublicKeyAuth` 用 sum-type（struct + 私有方法）允许：
- 构造时只填 `KeyID`（Profile 常见路径）
- 构造时只填 `Signer`（agent 转发、测试场景）
- 两种用法都由 sshclient 同一个方法处理

### 4. LRU cache for expensive objects
`signerCache` 是 per-connector 的 LRU，避免每次 Open 同一 key 都解析一次。容量 64 适合 95% 场景。
