## 关键决策

1. **`EventEmitter` 接口在 known_hosts 包内定义**，不 import wails runtime：
   - 核心层（known_hosts）零外部依赖，可独立单测
   - 接口方法 `EmitTrustRequest(ctx, TrustRequest)` 故意**只传必要字段**，
     避免 wails EventName / data interface{} 等 wails 特有概念渗透到核心层
2. **`wailsEmitter{}` 零字段 struct 同时实现两个接口**（`app.EventEmitter` + `knownhosts.EventEmitter`）：
   - 一个 struct 满足两个接口，避免 main.go 维护两份 emitter 适配器
3. **`EmitTrustRequest` 非阻塞**：emit 后立即返回，**不**等 reply。
   实际等 reply 走 `trustReplyCh` 另一条路径（select + 60s 超时）
4. **`trustReplyCh` 容量 1 + 5s `ReplyTrust` 超时**：
   - 容量 1：保证无挂起请求时立即返回（不阻塞），也避免过期 reply 堆积
   - 5s 兜底：异常（重复 reply / 无挂起请求）不永久阻塞
5. **ID 严格匹配**：拿到错 ID 的 reply 返回 mismatch error，让 SSH 握手失败（不静默放行）
6. **`len(ev.Data) > 0` 门控 sub:overflow 累加**：与 v0.2.3 同样的"防递归 drop 通知"设计
7. **TrustRequestModal 即使 `App.TrustHost` 返回 error 也强制关 modal**：
   让 SSH 连接走 timeout 失败路径，**不**让 UI 卡住

## 已知限制（v0.5.0 接受，v0.6+ 优化）

- `trustReplyCh` 容量 1：并发 SSH 连接到不同未知 host 时只有第一个能拿到正确 reply，
  其余会 timeout —— 实际场景（用户在 modal 里点之前不会同时点第二次连接）下不影响
- 真正的并发 trust 场景需要 `map[requestID]chan TrustReply`，留给 v0.6+

## 集成时的修复（主理自己干）

**两个测试逻辑 bug**：

1. `TestHostKeyCallback_TrustRequest_Timeout`：原测试同步调 `waitForCall` 在 `cb()` 之前，
   但 emit 是 `cb()` 内部发出 → 2s 内没信号。修：把 `waitForCall` 放 goroutine。
2. `TestReplyTrust_AfterRejection_StateRecovers`：`mockEmitter.waitForCall` 固定返回
   `calls[0]`，多次 emit 后拿到的是过期 ID → ID mismatch error。修：FIFO 队列语义
   （pop 最早一条）。

## 教训

- **接口放在消费方定义**：让 known_hosts 自己定义 `EventEmitter` 接口（不引用 wails）
  是最干净的方式，避免循环依赖 + 保持核心层可独立测试
- **零字段 struct 同时实现多个接口**是 Go 妙用 —— 一个 `wailsEmitter{}` 满足
  `app.EventEmitter` + `knownhosts.EventEmitter` 两个接口，main.go 不需要维护
  两份适配器
- **测试 mock 的 FIFO 语义**：测试 mock 应该模拟"真实事件流"的语义（顺序消费），
  不是简单"取第一个"——后者会在多次 emit 场景下产生过期数据
