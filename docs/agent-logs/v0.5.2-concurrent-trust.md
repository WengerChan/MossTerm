## 关键设计决策

### 1. per-request channel 优于单 channel 的 rationale

**旧设计（v0.5.0）的根本问题**：
- `trustReplyCh chan TrustReply` 容量 1 → 同一时刻只能承载一条 reply。
- 多个 SSH 连接同时触发 HostKeyCallback（典型场景：批量连接 5 个新 host）时：
  - 第一个等待者 select 拿到 reply，**消费掉**；
  - 第二个等待者拿不到（buffered channel 空了），只能 timeout；
  - 第三个等待者同理。
- v0.5.0 的"ID mismatch 错误"就是这个缺陷的症状 —— 第二个 callback
  拿到了**第一条** reply，reply.ID 必然不等于自己的 req.ID。
- dev-log 写"留给 v0.6+" 兑现 v0.5.2。

**新设计（v0.5.2）**：
- 每个 HostKeyCallback 独立 `make(chan TrustReply, 1)`，注册到
  `trustWaiters[req.ID] = replyCh`。
- ReplyTrust 用 ID 找对应 replyCh，**只**写到那一条 channel。
- 任意 N 个并发连接都能各自等待各自的 reply，per-request 隔离。

**为什么不选别的方案**：
- (A) 共享 channel + 显式 ID 路由：仍需要某个数据结构（map / queue）把
  reply 路由到对的 receiver；这就是新设计的本质，绕不开。
- (B) 全局 mutex 串行化所有 callback：并发退化为串行，破坏 x/crypto
  SSH handshake 的并发模型。
- (C) Buffered channel 容量 = N：不知道 N（用户随时新建连接），且
  容量浪费；map 是真正的"按需 N"。
- (D) 异步回调（replyCh 满了就丢）：破坏"必须等用户决定"的语义。

### 2. 锁粒度：独立 `trustMu`

不复用 `m.mu`（保护 entries 的 RWMutex）。理由：

- `m.Add` 持 `m.mu` 期间要 `os.OpenFile + WriteString`（文件 IO，~ms 级），
  不应被 trust map 的关键区阻塞。
- HostKeyCallback 的"已知 host 路径"只持 `m.mu.RLock` 读 entries，
  defer 删 waiter 只持 `trustMu` —— 两条路径完全解耦，**完全无竞争**。
- 锁顺序约束简单：永远不嵌套 `trustMu + m.mu`（写代码遵守，不依赖
  runtime 检测）。

### 3. defer delete waiter（无泄露）

每个 HostKeyCallback 的退出路径（trust / reject / unknown-action /
timeout）都通过 `defer func() { m.trustMu.Lock(); delete(...); m.trustMu.Unlock() }()`
清理 map。defer 是无条件执行的，不需要每个 return 点手动删。

验证：
- `TestHostKeyCallback_TrustRequest_TrustFlow` —— trust 路径后 `len(m.trustWaiters) == 0`
- `TestHostKeyCallback_TrustRequest_Timeout` —— timeout 路径后 `len(m.trustWaiters) == 0`
- `TestHostKeyCallback_ConcurrentTrustRequests` —— 5 个 callback 全部退出后 `len == 0`

### 4. ReplyTrust 5s 兜底 timeout

```go
select {
case ch <- TrustReply{ID: requestID, Action: action}:
    return nil
case <-time.After(5 * time.Second):
    return errors.New("knownhosts.ReplyTrust: reply channel full or handler deadlocked")
}
```

正常情况下 `ch` 容量 1 + receiver 在 select 里等 = send 立即成功。
5s timeout 永远不会触发，仅是"handler 死锁"的防御性兜底。

### 5. ID mismatch 不可能发生（v0.5.2 语义变化）

旧设计的"check `reply.ID != req.ID`"在新设计下不需要：
- replyCh 一一对应（每个 HostKeyCallback 一个）；
- ReplyTrust 用 requestID 找对应 replyCh 写入；
- callback 的 `<-replyCh` 拿到的 reply 必然是当初发给自己的。

`TestHostKeyCallback_TrustRequest_IDMismatch`（v0.5.0 写的）改成
`t.Skip()` 桩，并在注释里指向替代测试
`TestReplyTrust_UnknownID_ReturnsError`。

### 6. 同步（不开 goroutine）

HostKeyCallback **不开 goroutine**。SSH handshake 本身就是同步的
（x/crypto `ClientConfig.HostKeyCallback` 返回 error 决定 handshake
成败），开 goroutine 反而增加复杂度（要 spawn 等待 + 把结果回灌到
caller goroutine）。

并发通过 N 个独立 goroutine 调 N 个 callback 实现 —— 每个 callback
自己的 goroutine 自己阻塞在 select 里等 reply，相互独立。

## 行为差异（vs v0.5.0）

| 场景 | v0.5.0 行为 | v0.5.2 行为 |
|---|---|---|
| 单连接 trust 未知 host | OK（用唯一一条 replyCh） | OK（用专属 replyCh，语义不变） |
| 多连接并发 trust 多个 host | 只有第一个能 trust，其余 timeout | 全部能 trust，互不干扰 |
| ReplyTrust 用错 ID（waiter 在） | send 成功（buffered 写错 ID 的 reply），callback 拿错 ID → "ID mismatch" error | 立即 error（map 查找未命中），callback timeout |
| ReplyTrust 无 waiter | send 成功（buffered channel 默默吸收 reply） | 立即 error（map 查找未命中） |
| HostKeyCallback trust 路径 | trust 成功，channel 被消费 | trust 成功，replyCh 被消费 + defer 删 map |
| HostKeyCallback reject 路径 | reject 成功，channel 被消费 | reject 成功，replyCh 被消费 + defer 删 map |
| HostKeyCallback timeout 路径 | 60s 后 error，channel 没人消费 | 60s 后 error，defer 删 map |
| 已知 host 路径 | 跳过 trust，channel 不动 | 跳过 trust，map 不动 |
| MITM 路径 | 跳过 trust，channel 不动 | 跳过 trust，map 不动 |

**关键变化**：
- `ReplyTrust` 错误语义更准确（"找不到 waiter"立即 error，前端可重试 / 放弃）
- 并发 trust 场景从"v0.5.0 不支持"升级到"v0.5.2 完全支持"

## 踩坑 / 新发现

### 1. `TestHostKeyCallback_ConcurrentTrustRequests_MixedActions` 的隐藏 race

**问题**：第一版测试用 responder goroutine 的 i 作为 key 查 actions map，
但 emit 顺序不保证（responder[k] 拿到的可能是 callback[3] 的 emit，不是
callback[k] 的）。结果：trust 决策错路由。

**修法**：responder 根据**捕获的 host name** 决定 action（解析
"mixed3.example.com" 拿到 index 3，查 map）。这样无论哪个 callback 先
emit，决策都跟着 host 走。

**教训**：并发测试里**永远不要**用 goroutine index 当业务 key，必须
用 goroutine 实际捕获的请求数据（host / ID 等）做关联。

### 2. `TestReplyTrust_NoPendingRequest_ReturnsError` 的名字 vs 实际行为

v0.5.0 这个测试名字叫 "_ReturnsError" 但实际**断言不返回 error**（因为
旧版 buffered channel send 默默成功）。这导致：
- 测试名字误导
- 测试本质是"不阻塞 2s"而不是"返回 error"

v0.5.2 起新行为真的返回 error，测试名字终于和语义对齐。我把测试 body
更新成真断言"返回 error"。

### 3. uncommitted working tree 的预存失败

跑 `./...` 时发现 `internal/sshclient/integration_test.go` 的
`TestConnector_OpenSession_FullSFTPPath` 失败：
```
sess.Write n = 16, want 18
```

**根因**：测试 magic number 18 是错的 —— `"echo regression\n"` 是 16
字节（e-c-h-o-space-r-e-g-r-e-s-s-i-o-n-\n），不是 18。

**不在我范围**：这是 v0.5.2 spec 在 working tree 的另一个任务（sshclient
integration test），**和我无关**。我只在 /tmp/MossTerm_test 跑了我的
knownhosts 改动。

**给主理**：主理做总集时如果看到这个 fail，知道是另一个 task 的
test bug（"18" → "16"），不是 regression。

### 4. defer delete 写法的 race-free 验证

担心点：defer 在 callback 返回时执行，但 `m.trustMu.Lock()` 是阻塞的，
如果别的 goroutine 持锁会怎样？

**分析**：
- trustMu 只被 HostKeyCallback（注册/删 waiter）和 ReplyTrust（查找 waiter）两个地方用。
- 两个地方都是"短临界区 + 不持锁 IO"。
- 删 waiter 不会和别的 HostKeyCallback 注册 waiter 死锁（map 操作是
  几纳秒）。
- 删 waiter 不会和 ReplyTrust 查找 waiter 死锁（同样几纳秒）。

**实测**：`go test -race -count=5` 跑 5 遍，干净。

### 5. `MockEmitter` 容量

`newMockEmitter(N)` 用 buffer size = N，确保 N 个 emit 都 signal 成功
（不会因为 signal channel 满而丢失）。如果测试里有更多 emit 超过
bufSize 会丢 signal，但本测试最多 N 个 emit，N 个 responder goroutine
消费，1:1 配对。
