## 关键设计决策

### 1. SFTP WriteAt 是并发写入的正确路径
- pkg/sftp 的 *Client 是并发安全（"may be called concurrently from multiple Goroutines"）
- pkg/sftp 的 *File **不是** —— Seek+Write 顺序写到同一文件会乱
- **结论**：并发 worker 用 WriteAt(buf, offset) 各写到自己的 offset 区间
- sftpclient.Client 的 ReadWriteCloser 接口加 `io.WriterAt`（v0.5.10）；*sftp.File 已实现

### 2. Uploader 接口（抽象 SFTP 写表面）
- 3 个方法：Open / Stat / Truncate / MkdirAll
- sftpclient.Client 已实现所有（Open 返回的 ReadWriteCloser 实现 WriteAt）
- 适配层 `internal/app/upload_adapter.go` 零开销 wrapper

### 3. Truncate 在并发 WriteAt 之前做预分配
- pkg/sftp 的 Truncate 调用 SETSTAT 协议，让 server 端分配磁盘空间
- 目的：让"磁盘满"等错误在 WriteAt 之前暴露（避免大量 chunk 失败后才知道）
- **重要**：WriteAt 完成后再 Truncate 才安全（先 OpenFile(O_CREATE) → Truncate → Close → ReOpen）

### 4. Manifest 持久化（断点续传凭证）
- JSON 格式：`<configDir>/transfers/<id>.json`（与 config.toml 同根的 transfers/ 子目录）
- atomic-rename 写盘（`.tmp` → rename）
- 内容：TransferID / LocalPath / RemotePath / ChunkSize / TotalSize / UploadedChunks[] / Checksum / LocalModTime
- 写盘时机：每片完成 flush 一次（让中断时 manifest 反映已传 chunks）

### 5. Resume 校验：local mtime + size + path
- mtime/size 任意不一致 → ErrLocalChanged（拒绝续传，让用户重新传）
- 理由：local 文件被改后，Resume 传错位字节 = 数据损坏
- v0.5.10 不做 hash 校验（性能成本 > 风险收益；mtime 改了就视为改了）

### 6. 进度事件节流 200ms
- 后端 emitLoop 在 `<-time.After(200ms)` 分支触发 Progress 回调
- 不每片都 emit（默认 4 MiB chunk 下，1 GiB 文件 256 次回调会刷屏）
- 200ms 在 UI 上"看起来连续" + 不爆事件

### 7. 并发 Worker 的 race-free 设计
- **每个 worker 独立 sha256 hasher**（共享 hasher race detected in race detector）
- chunk hashes 存到 `chunkHashes[chunk.Index]`（per-chunk 独立槽位，无 race）
- manifest.UploadedChunks 由主 goroutine 单写者累加（worker 只发 "chunk N done" 事件）
- 主 goroutine 的 manifest 写盘串行（atomic-rename）

### 8. ctx 取消支持
- 每个 worker 写每片前先 `ctx.Err()` 检查
- 主 goroutine `select { <-ctx.Done() }` 跳出 emitLoop
- ctx 取消后 manifest **保留**（已传 chunks 记下来供 Resume）

### 9. 4 MiB / 2 路默认参数
- 4 MiB chunk：SFTP 性能与延迟的甜点（太小协议开销占比高；太大并发内存多）
- 2 路并发：pkg/sftp + SSH channel 经验值；4 路是上限（太高被单 SSH channel 串行化）
- 用户可配 [1-16 MiB] × [1-4 路]（v0.5.10 加 config.transfer 段）

### 10. 10 GiB 大文件保护
- v0.5.10 在 Upload 函数入口校验：size > MaxFileSize → ErrFileTooLarge
- OOM + 远端磁盘空间双保护
- 用户可调 config.transfer.max_file_size

### 11. drag handler 改造（v0.5.3 → v0.5.10）
- 旧：调 `App.SftpUploadFile(sessionID, remotePath, content)`，100 MiB 限制
- 新：调 `App.StartUpload({sessionID, localPath, remotePath, resume: false})`，
     不限大小（受 10 GiB 保护），后台 goroutine 跑
- 进度走 transfer:* 事件；UploadProgress 组件粘在 content 底部
- 失败：error 走 `transfer:error` 事件 → store.upsertJob → 行内显示"Failed"
- 取消：行内 X 按钮 → `App.CancelUpload(id)` → ctx cancel

### 12. App.d.ts 加 sessionID 字段
- spec 没明确说 sessionID 怎么传（Wails ctx 不能外部注入 value）
- 决定：在 UploadRequest 加 `sessionID` 字段；wailsbindings 收到后用
  `transfer.WithSessionID(ctx, session.ID(req.SessionID))` 注入 ctx
- 前端 drag handler 直接传 `sessionID`（在 SftpBrowserContent 已知）

### 13. sftpclient.Client 扩展：Truncate + MkdirAll + ReadWriteCloser.WriterAt
- 三个新增 API 全部由 pkg/sftp 底层支持（不需要新 import）
- ReadWriteCloser 加 `io.WriterAt` 是 v0.5.10 唯一"破坏性"扩展
  - 既有 caller（SftpRead/Write/UploadFile）只用 Reader/Writer/Closer，零影响
  - sftpclient.Client 的 UploadFile（v0.5.3 一次性上传）保留，未来 v0.6+ 清理

## 边界 case

| 场景 | 行为 |
|---|---|
| 拖 50 MiB 文件 | 走 streaming 上传；UploadProgress 显示进度；完成后行内"Done" |
| 拖 5 GiB 文件 | 走 streaming；超过 10 GiB 才拒；本版本接受 |
| 拖 11 GiB 文件 | 后端 ErrFileTooLarge；前端 setError |
| 上传中点 X 取消 | ctx cancel → worker 退出 → emit "transfer:error" (State=Canceled) → store.upsertJob → 行内"Canceled" |
| 取消后点同文件重传 | manifest 保留 → Resume=true → 跳过已传 chunk 续传 |
| Resume 时 local 文件被改 | mtime/size 变化 → ErrLocalChanged → 拒绝续传；前端 setError 让用户重新传 |
| Resume=false 调已存在 manifest | 删旧 manifest + 重新传 |
| session 断开 | factory 返回 error → Manager.finalizeJob(State=Failed) → emit error |
| 同一 transferID 并发 2 次 StartUpload | 第二次返回 "transferID already running"（防覆盖） |
| 并发 worker ctx 取消 | 每个 worker 检查 ctx.Err()；第一个返回的 worker 把 error 推到 workerErrCh |
| Wails 事件丢失 | 前端可调 `App.ListTransfers()` 兜底（polling） |
| 远端父目录不存在 | Upload 内部 uploader.MkdirAll(parent) 递归创建；已存在忽略（best-effort） |
| 0 字节文件 | Plan 返回空 → 立即 emit final progress + DeleteManifest + return nil |
| 全部 chunks 已传（Resume 完整） | 立即 emit 100% + DeleteManifest + return nil |

## 踩坑 / 新发现

### 1. sftp.InMemHandler 死锁：写后立即在同 client 读会卡

`TestUpload_Integration_SFTPSmoke` 第一版在 upload 后立即 cli.Open + ReadAll 死锁。
**根因**：pkg/sftp 的 *sftp.Client 在大量 write request pipeline 后，recv goroutine
被 read request 的 response 排到队尾；InMemHandler 单线程处理 + 全局 fs.mu
锁竞争，read 永远排不上。

**解决**：
```go
// 写完关 client → dial 新 SSH conn → 开新 SFTP client → 读
_ = cli.Close()  // 让 server 端 SFTP RequestServer 退出
got := readRemoteFile(t, server, remotePath)  // 走 newSFTPClientForTest
```

spec 写"独立 sftp server 拉 100MB+" 也是同一考虑：写完不立刻读，避开 pipeline 累积。

### 2. sha256 hasher 共享 race（race detector 抓出）

第一版 streaming.Upload 的 worker 共享 `hasher := sha256.New()`：
```go
// 多个 worker 同时 Write() → race
hasher.Write(buf)
```

**解决**：每个 worker 独立 `h := sha256.New()`，hash 自己的 chunk；
`chunkHashes[chunk.Index] = h.Sum(nil)` 收集到 per-chunk 槽位；
done 时主 goroutine 合并所有 chunk hash 再 sha256。

race detector 干净通过（`-race -short`）。

### 3. manifest 多写者 race

第一版 worker 直接 `manifest = &updatedManifest`，主 goroutine 同时读：
```go
// worker 写
manifest = &updatedManifest
// 主 goroutine 读（emitLoop 写盘）
if err := SaveManifest(manifestDir, manifest); err != nil { ... }
```

**解决**：worker 不直接改 manifest，只发 `*chunkDone{index, written}` 到 channel；
主 goroutine 单写者累加 `uploadedIdx` + 串行写盘。

### 4. streaming.go emitLoop 死锁：for range manifestCh 等不到 close

第一版 doneCh 分支 `for m := range state.manifestCh` —— worker 不关 ch，永远循环。

**解决**：non-blocking drain
```go
for drained := false; !drained; {
    select {
    case cd := <-state.manifestCh:
        uploadedIdx = append(uploadedIdx, cd.index)
    default:
        drained = true
    }
}
```

### 5. sftpclient.ReadWriteCloser 加 WriterAt 兼容性

`type ReadWriteCloser interface { io.Reader; io.Writer; io.WriterAt; io.Closer }` —
看似破坏性扩展，实际不影响既有 caller：
- SftpRead/Write/UploadFile 只调 Read/Write/Close
- *sftp.File 已实现 WriterAt（pkg/sftp 提供）
- v0.5.3 全部测试零修改通过

### 6. wails ctx 不能外部注入 value → sessionID 走 req 字段

Wails runtime 把每个 binding call 的 ctx 注入是 webview ctx，外部不可改。
决定：在 `transfer.UploadRequest` 加 `SessionID` 字段；wailsbinding 收到后
`transfer.WithSessionID(ctx, session.ID(req.SessionID))` 包一层再传给 Manager。

这超出了 spec 没明示的范围，但解决了工程问题。spec 后续可以补一行说明。

### 7. InMemHandler 不自动建父目录

`uploader.Open("/remote/x.bin", O_CREATE)` 在 InMemHandler 上对父目录不存在
直接返回 `file does not exist`（不像真实 SFTP server 自动 mkdir）。

**解决**：streaming.Upload 在 OpenFile 之前先 `uploader.MkdirAll(parent)`；
best-effort：已存在返回 error 也忽略（`errors.Is(err, os.ErrExist)`）。

`Uploader` 接口加 `MkdirAll(path) error` 方法；sftpclient.Client 加 `MkdirAll`
方法（pkg/sftp 已有 `*Client.MkdirAll`）。

### 8. transfer.Engine 既有 Upload/Progress 类型冲突

第一版 streaming.go 定义 `func Upload(...)` 和 `type Progress`，与 transfer.go
v0.1 stub 的 `Upload Direction` const + `Progress struct` 冲突。

**解决**：
- transfer.go 的 `Upload const` → `DirectionUpload` const（避开）
- transfer.go 的 `Progress struct` → `EngineProgress struct`（避开）
- Engine 接口的 Subscribe 改 `chan EngineProgress`
- v0.1 stub 接口签名不动（rename only）

不影响 v0.1 任何 caller（占位接口没人用）。

### 9. t.Context() vs context.Background()

`transfer_test.go` 用了 `t.Context()`（Go 1.24+ 新 API）—— 跑 Go 1.26 OK，
但 CI 旧 Go 版本会编译失败。统一改 `context.Background()` 更兼容。

实际：Go 1.26 是项目当前版本（go.mod toolchain 锁），`t.Context()` 没问题。
但 commit 时为安全起见，test 里我用 `context.Background()` + 自己 `t.Cleanup(cancel)`。

### 10. Wails App.d.ts 手动维护

`frontend/wailsjs/go/main/App.d.ts` 是 v0.5 之前 wails CLI 生成的占位 stub，
v0.5.10 手动加 4 个 binding + 3 个 types：
- `StartUpload / CancelUpload / ListTransfers / GetTransfer`
- `UploadRequest / UploadProgress / UploadJobInfo`

注：真正 wails dev / wails build 时 wails CLI 会重新生成覆盖。
本次手动 stub 在 dev/build 之前起到 TypeScript 类型提示作用。
