## 关键设计决策

### 1. 公开 `SftpClient` 包装私有 `sftpFor`

spec 标 `sftpFor` 为"私有方法"，但 `internal/ui/wailsbindings` 是不同 package，
无法访问私有方法。两个选择：

- (A) 改 spec，让 `sftpFor` 公开
- (B) 加一个一行公开 wrapper

选 (B)：保持 `sftpFor` 私有语义（内部实现），加公开 `SftpClient(sessID)` 作为 stable Go API。
Wails 反射的 *sftpclient.Client 含未导出字段，不会被自动 binding 到前端。

### 2. 懒加载 + 复用（Lazy + Cache）

session.MemoryManager 不持有 sftp client；sftpFor 第一次被调时懒构造。
后续命中同一 `map[session.ID]*sftpclient.Client` 条目。
sftp pkg 的 client 是 goroutine unsafe（sftpclient 文档明确），复用避免每次开/关 subsystem channel。

### 3. 生命周期 = Lazy Eviction（spec 推荐方案）

session 关闭（state → Closed）→ 下次 sftpFor 看到 state≠Established → 删 map 条目 + 返回 error。
wailsbindings 拿到 error 后让前端关闭文件浏览器。
优势：不用 hook session 关闭事件、不用双簿记（Manager 关 + App 关）。

### 4. 并发安全：double-checked locking

```go
a.sftpMu.Lock()
cached, ok := a.sftpClients[sessID]
a.sftpMu.Unlock()
// 1) 命中
if ok && cached != nil {
    // 验 session state（不持 sftpMu 时 Get，避免死锁）
    if state != Established {
        delete + close + return error
    }
    return cached
}
// 2) miss：验 state → 拿 connector → type assert → RawClient → openSftp
client, err := a.openSftp(sshClient)
// 3) 写回 map（持锁内**二次检查**，处理"两个 goroutine 都 miss"）
a.sftpMu.Lock()
if existing, ok := a.sftpClients[sessID]; ok && existing != nil {
    a.sftpMu.Unlock()
    _ = client.Close()  // 关闭 wasted open
    return existing, nil
}
a.sftpClients[sessID] = client
a.sftpMu.Unlock()
return client, nil
```

> **trade-off**：两个 goroutine 同时 miss 时会浪费一次 openSftp（被二次检查关掉）。
> 严格 first-write-wins 需要 `sync.Once` 之类；v0.5.1 spec 接受"wasted open"。

### 5. openSftp 钩子（小测试性改造）

```go
type App struct {
    // ...
    openSftp func(*ssh.Client) (*sftpclient.Client, error)
}

// New(deps):
a.openSftp = func(c *ssh.Client) (*sftpclient.Client, error) {
    return sftpclient.Open(c)
}
```

**动机**：sftpclient.Open 内部要 `*ssh.Client.NewSession()` + `RequestSubsystem("sftp")`，
需要真 SFTP server。v0.5.1 unit test 只验证 app 层的 map 生命周期（懒加载/复用/lazy evict/App.Close），
不该为这个而启动 sftp.NewServer。

测试通过 `a.openSftp = mockFactory` 注入，省掉真 SFTP server。

### 6. 测试基础：in-process SSH server

为了测试 sftpFor 的真路径（需要 c.client ≠ nil），写了一个最小 SSH server
（`internal/app/sftp_test.go::sshTestServer`）：

- 接受任何用户名/密码
- ed25519 临时 host key
- 接受 "session" channel + 全部 request 回复成功（pty-req / shell / env 等）
- ctx 取消时主动 `sconn.Close()`，让 `sconn.Wait()` 不卡

完整 sftpFor 流程：mm.Open → factory → *sshclient.Connector → Dial 真握手 → OpenSession
（拿 pty / 拿 stdin pipe / shell）→ state=Established → sftpFor 走通。

## 顺手修的 v0.1 隐 bug

`internal/sshclient/client.go` OpenSession **一直把 StdinPipe/StdoutPipe 写在 Shell() 之后**。
x/crypto v0.22.0 的 `Session.StdinPipe` 检查 `s.started == true` 就返回
`"StdinPipe after process started"` —— 而 `Shell()` 内部 `s.start()` 会把 `started` 置 true。

```go
// 错的顺序（v0.1 - v0.5.0）：
Shell()         // s.started = true
StdinPipe()     // 报错：s.started == true
StdoutPipe()    // 同上

// 修正（v0.5.1）：
StdinPipe()     // 设 s.stdinpipe = true
StdoutPipe()    // 设 s.stdoutpipe = true
Shell()         // s.start() 检查 s.stdinpipe / s.stdoutpipe，
                //   已设则跳过 default setupFd
```

**为什么 v0.5.0 之前没爆**：
sshclient 单元测试用 `keepalive_test.go` mock，不调 OpenSession。
没有真 SSH server 的 integration test。
本次 v0.5.1 写 in-process SSH server 第一次触发到。

## 8 个 binding（实际 7 个）

spec 头部写"暴露 8 个 SFTP binding"，但代码列表只列了 7 个。
`frontend/wailsjs/go/main/App.d.ts` 也是 7 个 —— 我跟齐了 7 个。
"8" 应该是 spec typo。

| # | 签名 | 调底层 |
|---|---|---|
| 1 | `SftpList(ctx, sessionID, path string, pageSize int, pageToken string) (sftpclient.ListPage, error)` | `client.List(...)` |
| 2 | `SftpStat(ctx, sessionID, path string) (sftpclient.Entry, error)` | `client.Stat(path)` |
| 3 | `SftpMkdir(ctx, sessionID, path string) error` | `client.Mkdir(path)` |
| 4 | `SftpRemove(ctx, sessionID, path string) error` | `client.Remove(path)` |
| 5 | `SftpRename(ctx, sessionID, oldPath, newPath string) error` | `client.Rename(o, n)` |
| 6 | `SftpRead(ctx, sessionID, path string) ([]byte, error)` | `client.Open(p, O_RDONLY)` + `io.ReadFull` ≤1 MiB |
| 7 | `SftpWrite(ctx, sessionID, path string, data []byte) error` | `client.Open(p, O_WRONLY\|O_CREATE\|O_TRUNC)` + `f.Write` |

**Wails 反射规则**（api.go 顶部注释已写清）：
- `context.Context` 必须第一个参数
- `[]byte` → 前端 `Uint8Array`
- `time.Time` → RFC3339 字符串
- `os.FileMode` (uint32) → number
- 错误返回 → 前端 `.catch()` 捕获

**SftpRead 上限**：1 MiB（v0.5.1 spec：仅支持小文件）。超出用 v0.5.2+ 的 transfer.Engine 流式下载。

## 踩坑 / 新发现

1. **OpenSession Shell/StdinPipe 顺序 bug（v0.1 起就错）**：
   见上文"顺手修的 v0.1 隐 bug"。强烈建议主理在合 v0.5.1-A 时单独起一个 v0.5.1-B 补一个
   sshclient 真 OpenSession 的 integration test 守住这条路径（避免以后 regression）。

2. **in-process SSH server 不是 plug-and-play**：
   - 必须 accept "session" channel + 全部 request（pty-req / shell / env / window-change / subsystem）
     回复 true；否则 *sshclient.Connector.OpenSession 卡在 SendRequest
   - 必须 ctx 取消时主动 `sconn.Close()`，否则 `sconn.Wait()` 永不返回，test cleanup 卡死
   - 必须 keepalive 也要回复（client 30s tick 触发 SendRequest），否则 v0.5.2+ 长测试会暴露问题
   - 第一次跑全测试时因为 StdinPipe/Shell 顺序 bug 卡住发现，给出 50 行左右堆栈才看出

3. **双 check 锁的 wasted open（spec 接受）**：
   10 goroutine 并发时 sftp open 计数可能 > 1（被二次检查关掉的"wasted"实例）。
   spec 接受；想严格 0 wasted 需要 `sync.Once` per session 锁。下次有人抱怨浪费可以加。

4. **保留 sftp 客户端 map 用具体类型 *sftpclient.Client**：
   spec 明确"懒加载，session 关闭时移除" + `map[session.ID]*sftpclient.Client`。
   想加 Close mock 必须用 interface，破坏 spec 写法。妥协：App.Close 测试用"map 清空"
   间接断言 Close 被调（"len==0 + closeSftpClients 源码：if c==nil continue; else c.Close()"）。

5. **spec 写"8 个 binding"**：
   实际列 7 个，frontend d.ts 也 7 个。`openSftp` 错误信息加了"spec 草稿写 8 个，
   实际列 7 个"备注，给后任查阅。
