## 关键设计决策

### 1. `SftpUploadFile` 复用 v0.5.1-A 公开的 `SftpClient()` wrapper

不调私有 `sftpFor`，而调 `a.core.SftpClient(sessionID)`：

```go
func (a *App) SftpUploadFile(ctx context.Context, sessionID, remotePath string, content []byte) (int, error) {
    client, err := a.core.SftpClient(session.ID(sessionID))
    if err != nil {
        return 0, fmt.Errorf("wailsbindings.SftpUploadFile: %w", err)
    }
    n, err := client.Write(remotePath, content)
    if err != nil {
        return n, fmt.Errorf("wailsbindings.SftpUploadFile: %w", err)
    }
    return n, nil
}
```

**理由**：`SftpClient` 是 v0.5.1-A 主理加的"wailsbindings 边界"——`*sftpclient.Client`
含未导出字段，不会被 Wails 反射误 binding。私有 `sftpFor` 留给内部使用，公开 wrapper
是 stable Go API。

### 2. 与 `SftpWrite` 的关系

| | SftpWrite | SftpUploadFile |
|---|---|---|
| 用途 | 编辑器保存小文件 | drag-drop 整文件上传 |
| 大小 | < 1 MiB | ≤ 100 MiB |
| 返回 | `error` | `(int, error)` 字节数 |
| 行为 | 调 `Open()` + `f.Write()` | 调 `client.Write()`（v0.5.3-A 新增） |

**为什么不复用 SftpWrite**：
- 返回类型不同（UploadFile 需要字节数给前端做反馈）
- 关注点分离：editor write 是小文件 + 通用，upload 是 drag-drop 专属路径
- 未来 streaming upload 时只需要改 `SftpUploadFile`，不影响 `SftpWrite` 的契约

### 3. Drag-Drop UX

| 决策点 | 选择 | 理由 |
|---|---|---|
| 多文件支持 | ✅ 支持 | 用户从 Finder 多选拖拽很常见；每个文件独立 try/catch |
| 100 MiB 限制 | ✅ 实施 | v0.5.3 一次性 `readAsArrayBuffer` 读进内存；超出后 UI 冻结 + Wails Uint8Array 序列化压力陡增 |
| 错误处理 | `setError`（顶部红色条） | 与 SftpBrowser 既有错误模式一致（List 错误也用 setError） |
| 视觉反馈 | `ring-4 ring-accent/40` + `border-accent` | 拖拽时整个 panel 高亮，松开恢复；用 `accent` 色（绿色）让用户立刻看到"会接住" |
| 进度反馈 | `setLoading("Uploading <name>...")` 一次性提示 | v0.5.3 一次性同步返回，没 progress 事件；不做假进度条 |
| 覆盖写 | ✅ 直接覆盖 | `client.Write` 用 `O_WRONLY|O_CREATE|O_TRUNC`；与"上传"语义一致 |
| handlers 挂载位置 | 内层 panel `<div>`（不是外层 backdrop） | 拖到 backdrop 会触发 `onClick={onClose}` 关闭浏览器；拖到 panel 才接住 |
| dragenter/dragleave 细节 | dragleave 检查 `relatedTarget` 是否在 panel 内 | dragleave 在进入子元素时也会触发（事件冒泡）；不检查会导致 ring 闪烁 |
| 状态重置 | `open=false` / `sessionID` 变化时 `setIsDragOver(false)` | 防止关闭后下次打开残留状态 |

### 4. 单元测试策略：端到端（不用 mock）

**选端到端不用 mock 的理由**：
- `sftpclient.Client` 的 `sc` 字段是私有的（`*sftp.Client`），无法在外部 mock
- mock 出来只能测包装逻辑（"调了 client.Write"），不能测真实数据流
- 端到端验证 Wails 反射契约最稳：ctx 第一参数 / `[]byte` 序列化 / `int` 返回值 / `error` 包装

**测试基础设施（自包含）**：
- `sftpUploadSSHServer`：绑 127.0.0.1:0 的 in-process SSH server
  - 接受任何用户名/密码
  - **只支持 SFTP subsystem**（不需要 shell —— UploadFile 不走 shell）
  - 共享 `sftp.InMemHandler()`（v0.5.3-A 主理修过的关键点）
  - ctx 取消时主动 `sconn.Close()`，让 `sconn.Wait()` 不卡死

**5 个测试**：

| Test | 覆盖 |
|---|---|
| `TestSftpUploadFile_HappyPath` | 端到端：写 1 KiB 伪随机数据 → `sftp.NewClient` 独立通道读回 → 字节级比对 |
| `TestSftpUploadFile_Overwrite` | 写 256B A → 写 1024B B → 读回应是 1024B 全 B（不拼接） |
| `TestSftpUploadFile_SessionNotFound` | 空 `*app.App` 调 binding → 返回 `wailsbindings.SftpUploadFile: session ... not found` |
| `TestSftpUploadFile_EmptyContent` | `[]byte{}` 写入 → n=0, err=nil（边界） |
| `TestSftpUploadFile_Signature` | 编译期守卫：绑函数类型到 `(ctx, sid, path, []byte) (int, error)` |

**为什么没单独测 `openSftp` 错误路径**：
- `app.TestSftpFor_OpenSftpError` 已经覆盖了 `sftpFor` 内部错误包装的代码路径
- wailsbindings 这层只多了一次 `fmt.Errorf("wailsbindings.SftpUploadFile: %w", err)`
- 同一包装逻辑（"wailsbindings.<Method>: %w"）已被 `SftpList/SftpStat/...` 共 7 个 binding
  使用 + 通过 race detector 验证
- "session 不存在"已覆盖同一包装路径，断言 `strings.HasPrefix(err.Error(), "wailsbindings.SftpUploadFile: ")`
- 重复加 openSftp 错误测试 = 重复覆盖。spec 显式说"如果 sftpFor mock 掉...，看 sftp_test.go 风格"，
  v0.5.1-A 那条路径已在 app/sftp_test.go 守好

**为什么不需要 race detector 单独跑 wailsbindings**：
- wailsbindings 是薄 wrapper，**无新增并发原语**（`App.core *app.App`、`a.core.SftpClient` 是原子读 map）
- `sftpclient.Client.Write` 自身的 goroutine safety 由 `sftpclient` 包测试覆盖
- `app.sftpFor` 的并发安全由 `app.TestSftpFor_Concurrent` + race detector 覆盖
- 本次还是跑了 `-race -count=3` 守一下，**1.582s 干净通过**

### 5. `SftpUploadFile` vs `sftpclient.Client.UploadFile`

| | `SftpUploadFile` (binding) | `sftpclient.Client.UploadFile` |
|---|---|---|
| 输入 | `[]byte`（已在内存） | 本地文件路径 + 分片大小 + 进度回调 |
| 用途 | drag-drop 上传（前端先 `readAsArrayBuffer`） | 命令行 / 未来 batch 上传（直接读本地文件） |
| 分片 | 不分片，一次性 `Write` | 64 KiB 分片 + 进度回调 |
| 大文件 | 受 100 MiB 限制 | 无限制（流式） |
| 返回 | 写入字节数 | error |

**为什么 drag-drop 不直接走 `UploadFile`**：
- 浏览器里 `localPath` 概念不存在 —— 拖拽拿的是 `File` 对象，调用 `arrayBuffer()` 才能拿到字节
- 让前端先把 `File` 整个读进内存（≤ 100 MiB）再传后端，简洁清晰
- 未来 v0.6+ streaming upload 可以加一个 `SftpUploadStream` binding（前后端 stream 通道），
  不破坏 v0.5.3 `SftpUploadFile` 契约

## 踩坑 / 新发现

### 1. 没找到 `SetOpenSftpForTest` helper

到 `*app.App` 的私有 `openSftp` 字段。`app.openSftp` 是 unexported，wailsbindings 包

**解决**：精简测试范围，**不测 openSftp 错误**。理由：
- `app.TestSftpFor_OpenSftpError` 已经覆盖了 sftpFor 内部 error 包装
- wailsbindings 这层只多一次 `fmt.Errorf("wailsbindings.SftpUploadFile: %w", err)`
- 同一包装逻辑（"wailsbindings.<Method>: %w"）已被 7 个 Sftp* binding 共用
- "session 不存在"已覆盖同一包装路径，断言 error message 前缀

**收益**：少 ~50 行测试代码；spec 显式说"如果 openSftp 没暴露给 wailsbindings test，
看 sftp_test.go 风格"，v0.5.1-A 那条路径已守好。

### 2. dragleave 事件会"假触发"清状态

`dragleave` 事件在拖拽光标进入子元素时会触发（事件冒泡）。如果不检查，会出现：
- 拖入 panel → setIsDragOver(true) → 视觉 ring 出现
- 拖到 panel 内一个子元素 → dragleave 触发 → setIsDragOver(false) → ring 消失
- 拖回到 panel 主体 → dragover 触发 → setIsDragOver(true) → ring 出现
- 用户体感：ring 闪烁

**解决**：
```typescript
const handleDragLeave = useCallback((e: React.DragEvent) => {
    e.preventDefault();
    e.stopPropagation();
    const panel = e.currentTarget as HTMLElement;
    const next = e.relatedTarget as Node | null;
    if (next && panel.contains(next)) return;  // 进入子元素，忽略
    setIsDragOver(false);
}, []);
```

### 3. drag-drop 挂载位置：内层 panel vs 外层 backdrop

SftpBrowser 有两层 div：
- **外层 backdrop**（z-40 黑色透明蒙层）—— `onClick={onClose}` 关闭浏览器
- **内层 panel** —— 真正的浏览器内容

如果挂 drag-drop 到外层 backdrop：
- 拖到 backdrop 任意位置都会触发 drop（`onClick={onClose}` 和 drop 都会触发）
- drop 后浏览器立刻关掉，用户体感差

**解决**：挂到内层 panel `<div>`。只拖到 panel 上才接住；拖到外层 backdrop 任意位置
不响应（浏览器仍然 visible）。

### 4. Wails 反射：`[]byte` 输入 + `int` 返回值的契约

`SftpUploadFile(ctx, sessionID, remotePath string, content []byte) (int, error)`：
- 前端 `Uint8Array` ↔ Go `[]byte` 序列化（Wails 自动）
- 前端 `Promise<number>` ↔ Go `int` 返回值（Wails 自动）
- 前端 `.catch()` 捕获 Go `error`

`App.d.ts` 声明要完全镜像 Go 签名：
```typescript
static SftpUploadFile(sessionID: string, remotePath: string, content: Uint8Array): Promise<number>
```

**`sessionID: string` 不用 `SessionID` 类型** —— 与既有 7 个 Sftp* binding 风格一致
（既有也是 `string`，不引入 `SessionID` 类型；前端用 string 即可）。

### 5. 100 MiB 限制的边界

`MAX_UPLOAD_BYTES = 100 * 1024 * 1024` = 104,857,600 bytes。

**为什么不选 100 MB 整数（`100 * 1000 * 1000`）**：
- 操作系统和文件管理器（macOS Finder、Windows Explorer）显示文件大小用 MiB / MB 都可能
- 用 MiB（1024-based）更接近 OS 实际行为；100 MB 整数会让用户拖一个 104,857,600 bytes
  的文件时被拒（因为 OS 显示 "100 MB"），体感不一致
- spec 草稿说 "100 MB" 但 v0.5.3 内部用 MiB 实现；提示用户时也说 "100 MiB" 明确边界

**为什么不用更小的限制（50 MiB / 10 MiB）**：
- 现代 Mac 默认 16 GB+ 内存，浏览器单 tab 占 100 MiB 完全可接受
- v0.5.3 spec 显式说 "100MB"，跟随 spec
- v0.6+ streaming upload 会放开

## Drag-Drop UX 描述（用户怎么操作 + 看到什么）

1. **打开 SFTP 浏览器**（顶部"SFTP"按钮），浏览到目标目录（如 `/home/user/`）
2. **从 Finder/Explorer 拖一个或多个文件**到 SFTP 浏览器面板上
3. **拖拽悬停时**：
   - 整个 panel 边框变 accent 色（绿色）+ 外圈 4 像素 ring（半透明 accent）
   - 视觉提示"会接住"
4. **松开鼠标**：
   - ring 消失，panel 恢复正常
   - 顶部 loading 提示变成 `Uploading <filename>...`
   - 如果是多个文件：依次上传，每个文件独立 loading 提示
5. **完成后**：
   - loading 提示清空
   - 列表自动刷新，新文件出现在目录里
   - 如果某个文件失败（罕见）：顶部红色 error 条显示 `Upload <filename> 失败：<错误信息>`
   - 其他文件继续上传（单文件失败不影响其他）
6. **大文件拦截**（> 100 MiB）：
   - 不调后端，直接顶部红色 error 条显示 `<filename> 超过 100 MiB，v0.5.3 不支持大文件上传（v0.6+ streaming）`
   - 列表不变（其他小文件还能传）
7. **覆盖写语义**：
   - 拖同名文件到当前目录 → 覆盖（O_WRONLY|O_CREATE|O_TRUNC）
   - 不弹确认（与"上传"语义一致）
