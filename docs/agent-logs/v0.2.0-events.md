# v0.2.0 · 后端工程师 task（events 批处理）

**派活时间**：2026-07-13  
**目标**：解决 v0.1.1 以来一直存在的 events 通道在高速数据下丢事件问题

## 任务描述

- `readLoop` 重写为 reader goroutine + main loop 双层结构
- 16ms ticker + 64KB accumulator
- 新增 `EventTypeOverflow` 事件 + `Event.OverflowBytes` 字段
- `tryPublish` 累计 overflow 字节数（不再静默丢）
- `fanoutLoop` 旁路 emit overflow（不走 events 通道）
- 6 个单元测试

## 关键设计决策

- **双层 goroutine**：reader 持 sess 引用 + dataCh(cap=8) → main loop + accumulator
- **overflow 累计语义**：events 满时整批丢，累加字节数；state/exit/error 不计
- **fanoutLoop 旁路 emit**：overflow 事件不走 events 通道（避免火上浇油）
- **顺序**：StorePending(false) → Swap(0) → if bytes>0 broadcast

## 吞吐预算

| 场景 | 吞吐 |
|---|---|
| 稳态（每 tick 一次 maxAccum） | 64KB / 16ms ≈ **4 MB/s** |
| 理论上限 | 256 MB/s |
| 典型 SSH 终端 | < 100 KB/s |

## review 发现（小弟顺手修 bug）

v0.1.4 的 `tryPublish` 在关停时**1/3 概率误报 overflow**：
- Go runtime 在多 case select 中均匀随机选
- 当 `s.done` 已关闭 + `s.events` 满时，1/3 选到 default
- 修复：default 分支加内层 `select { case <-s.done: ... default: ... }` 区分
- 回归测试：`TestTryPublish_SessionClosed_DoesNotIncrement`
